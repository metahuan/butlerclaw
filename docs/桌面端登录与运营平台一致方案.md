# 桌面端登录与运营平台一致方案

## 1. 要不要做登录？

**建议：要做，且与运营平台共用同一套账号与会员体系。**

| 原因 | 说明 |
|------|------|
| 会员与收费一致 | 修复功能、付费技能等需要「是否已登录 + 是否专业版」；若桌面不登录，只能一律当未登录/免费，体验割裂。 |
| 一处账号，多端通用 | 用户在 Hub 注册/登录后，在桌面用同一账号登录，会员、购买记录、身份（user/developer）一致，无需两套体系。 |
| 安全与合规 | 付费、权益校验以服务端为准；桌面只存 token，不存密码，不重复实现一套用户表。 |

因此：**桌面端应提供「登录 ButlerClaw 账号」能力，且账号、会员、权限与运营平台（Hub）完全一致。**

---

## 2. 与运营平台一致的含义

- **同一套用户数据**：用户表在 Hub 云数据库 `users` 集合，桌面不单独建用户库。
- **同一套鉴权**：使用 Hub 的 `user-api` 登录接口返回的 JWT；桌面请求 Hub 任意接口时带 `Authorization: Bearer <token>`。
- **同一套会员与权限**：`membershipLevel`（free/pro）、`membershipExpireAt`、`role`（user/developer）均以 Hub 的 `getUserInfo` / `GET /api/auth/me` 为准；桌面只读、不写。

实现方式：**桌面只做「登录客户端」**——调用 Hub 的登录接口拿到 token，本地仅持久化 token 与缓存的用户/会员信息，所有「这人是谁、是不是 pro」都通过请求 Hub 或使用缓存（来源于 Hub）得到。

---

## 3. 推荐方案概览

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **A. 桌面直调 Hub 登录 API** | 桌面内嵌登录框（邮箱+密码），POST Hub 的 `/api/auth/login`，拿 token 存本地 | 实现简单、无需改 Hub、与现有 Hub 登录完全一致 | 桌面需处理密码输入（可做安全措施：不明文日志、仅内存使用） |
| **B. 浏览器登录 + 回调传 token** | 桌面打开系统浏览器到 Hub 登录页，登录成功后 Hub 跳转 `butlerclaw://auth?token=xxx` 或本地回调页把 token 给桌面 | 密码不经过桌面、用户习惯与网页一致 | 需 Hub 支持回调（自定义 scheme 或 localhost 回调页）、桌面监听 scheme 或本地服务 |

**推荐优先做 A**，后续如需「不在桌面输密码」再补 B。

---

## 4. 方案 A：桌面直调 Hub 登录 API（推荐）

### 4.1 流程

1. 用户在桌面点击「登录 ButlerClaw 账号」→ 弹出登录窗口（邮箱、密码）。
2. 桌面端将 `POST {hubBaseUrl}/api/auth/login`，body `{ "email": "...", "password": "..." }`。
3. Hub 返回 200 时 body 含：`token`、`userId`、`username`、`membershipLevel`、`membershipExpireAt`、`role` 等（与现有 Hub 前端一致）。
4. 桌面将 token 及需要展示的字段写入 `~/.openclaw/hub.json`（或 openclaw.json 的 `hub` 节点），文件权限设为仅当前用户可读（如 0o600）。
5. 之后所有需要「已登录」或「会员」的请求（如技能下载、修复权限）均带 `Authorization: Bearer <token>`；会员状态可用登录时返回的字段做缓存，并定时或按需调 `GET /api/auth/me` 刷新。

### 4.2 与运营平台的一致性

- **账号**：同一 Hub 用户表、同一登录接口，桌面与网页共用。
- **Token**：同一 JWT，Hub 的 skill-api getDownloadUrl、auth/me 等已支持 Bearer，桌面直接复用。
- **会员**：`membershipLevel` / `membershipExpireAt` 来自 Hub 返回或 `/api/auth/me`，桌面不自己算、不自己存「会员类型」，只读并缓存。

### 4.3 安全要点

- 密码仅存在于内存和请求体，不落盘、不写日志。
- `hub.json` 仅存 token 与必要展示字段，权限 0o600；生产环境可考虑用系统密钥链存储 token（后续优化）。
- 若需在界面显示「已登录为 xxx」，只显示 nickname/username，不显示邮箱或敏感信息（按需脱敏）。

---

## 5. 方案 B：浏览器登录 + 回调传 token（可选增强）

适用于希望「从不在桌面输入密码」的场景。

### 5.1 流程

1. 桌面点击「登录」→ 打开系统默认浏览器，访问 Hub 的专门登录回调页，例如：  
   `https://<hub域名>/auth/desktop-callback?redirect=butlerclaw://auth`  
   或带 state 防 CSRF。
2. 用户在浏览器完成 Hub 登录（与网页一致）；登录成功后 Hub 跳转到 `butlerclaw://auth?token=xxx`（或 `http://127.0.0.1:port/callback?token=xxx`）。
3. 桌面端注册自定义 URL Scheme `butlerclaw://`（或起一个本地 HTTP 服务接收 127.0.0.1 回调），收到 token 后写入 `hub.json`，并可选调 `GET /api/auth/me` 拉取会员信息。
4. 关闭浏览器或提示「登录成功，请回到桌面」。

### 5.2 Hub 需配合

- 提供「桌面登录回调」页：完成登录后把 token 以 query 或 fragment 传给 redirect；或在该页用 postMessage + 本地小服务取回 token。
- 若用 `butlerclaw://`：需在桌面应用注册该 scheme（Windows 注册表、macOS Info.plist 等）。

### 5.3 与运营平台一致

- 登录仍在 Hub 完成，用户数据与 token 仍来自 Hub，桌面仍只存 token 与缓存；一致性同方案 A。

---

## 6. 桌面端建议实现内容（与运营平台一致）

无论采用 A 或 B，桌面端建议统一做以下内容，保证「登录体系与运营平台一致」：

| 项目 | 说明 |
|------|------|
| **登录入口** | 设置页或主界面提供「登录 ButlerClaw 账号」；已登录时显示「已登录为 xxx · 免费/专业版」及「退出」。 |
| **本地存储** | `~/.openclaw/hub.json`：`token`、`userId`、`username`、`membershipLevel`、`membershipExpireAt`、`updatedAt`；权限 0o600。 |
| **登录实现** | 方案 A：调用 `POST /api/auth/login`，写入 hub.json；方案 B：监听 scheme 或本地回调，拿到 token 后写入并可选调 `/api/auth/me`。 |
| **会员刷新** | 启动时若存在 token，先读缓存再异步 `GET /api/auth/me` 更新；使用修复/付费技能前可再刷新一次（如缓存超过 5–10 分钟）。 |
| **请求带 token** | 所有请求 Hub 的接口（技能下载、auth/me）统一在 HTTP 头带 `Authorization: Bearer <token>`；401 时清空本地登录态并提示重新登录。 |
| **退出** | 删除或清空 hub.json 中的 token 及相关字段；界面回到未登录状态。 |

这样，桌面端没有「自己的」用户系统，只有「Hub 登录态在桌面上的缓存与使用」，与运营平台在账号、会员、权限上完全一致。

---

## 7. 小结

- **桌面要做登录**，且**与运营平台一致**：同一账号、同一 token、同一会员与权限。
- **推荐方案**：先实现 **方案 A（桌面直调 Hub 登录 API）**，实现简单、与现有 Hub 完全兼容；若后续需要「不在桌面输密码」，再增加 **方案 B（浏览器登录 + 回调传 token）** 作为可选路径。
- 桌面端仅作为「登录客户端」：不维护用户表、不校验密码，只负责把 Hub 的登录结果（token）存好、并在请求 Hub 时带上，从而实现与运营平台一致的登录与会员体验。
